Solution Design · v1

Gestionnaire de CRA

Architecture de la v1 — tunnel CRA → validation/signature → facture → règlement, multi-organisations, prestataire en direct.

Document compagnon de ARCHITECTURE-SPINE.md · 29 juin 2026 · statut final

Ce document déroule la spine d'architecture : le contrat d'invariants qui empêche le front Nuxt, les modules NestJS et le schéma MariaDB — construits séparément — de diverger. La stack est posée ; le code possède le détail dès qu'il existe. Chaque décision load-bearing (AD) porte ce qu'elle lie, la divergence qu'elle empêche, et sa règle applicable.

Contexte

Outil web qui dématérialise la chaîne administrative d'une mission de prestation. Le problème chiffré : un délai de paiement négocié de 7 jours, mais ~15 jours d'allers-retours papier ; un couac 1 mois sur 3. La v1 vise un prestataire indépendant en direct avec son client, dans un cadre multi-organisations. Mantra produit, porté jusque dans l'archi : « simple, avec une vue rapide ».

Le produit a une valeur juridique (la validation client vaut signature électronique, Option A) : c'est ce qui justifie la rigueur sur l'immuabilité, le snapshot figé, l'audit transactionnel et la sauvegarde.

Paradigme & topologie

Architecture hexagonale stricte (ports & adapters), un module NestJS par contexte borné, derrière un Backend-For-Frontend Nuxt (Nitro). Le navigateur ne parle qu'au serveur Nuxt, qui termine l'authentification et relaie vers une API NestJS non publique. Toutes les dépendances pointent vers le domaine.

Les 8 contextes bornés sont des modules NestJS étanches : iam, projects, cra (le cœur), invoicing, settlement, notifications, audit, documents. Ils communiquent par événements de domaine sur un bus in-process — jamais en tapant dans le repo l'un de l'autre.

Invariants & règles

24 décisions d'architecture. ADOPTED = arbitrage déjà tranché (par le PRD, la réalité o2switch, ou un choix explicite en coaching).

Topologie & paradigme

AD-1 Topologie BFF

Empêche
que le navigateur appelle NestJS en direct ; un token côté client ; deux contrats à synchroniser.
Règle
Le navigateur ne contacte que les server routes Nuxt/Nitro. Le BFF est un proxy mince : termine l'auth, relaie, ne reshape pas. Un seul contrat fait foi, côté NestJS.

AD-2 Hexagonal strict

Empêche
que la règle métier fuie dans les services ou l'ORM ; des invariants juridiques contournables.
Règle
Domaine pur (sans décorateur framework/ORM) + ports + use-cases + adapters. Les dépendances pointent uniquement vers le domaine.

AD-3 Contextes bornés étanches

Empêche
deux propriétaires d'une même donnée ; un module lisant la table d'un autre.
Règle
Un contexte n'accède jamais au repo d'un autre. La donnée d'autrui se consomme via un port, ou via un événement de domaine.

Événements & cohérence

AD-4 Communication par événements de domaine

Empêche
le couplage du cœur à ses consommateurs.
Règle
Bus in-process (CraValidé, FactureDéposée…). Le cœur émet sans connaître ses abonnés.

AD-5 Garantie de livraison différenciée

Empêche
une trace d'audit perdue ; un échec d'email annulant une transition.
Règle
Dispatch dans l'Unit of Work. Audit = abonné synchrone, même transaction. Notifications via l'outbox transactionnel (AD-22).

AD-21 Statut affiché = projection d'événements

Empêche
deux contextes composant un statut divergent ; l'écriture croisée du statut.
Règle
Chaque contexte possède sa tranche (CRA→Validé ; Invoicing→Facturé ; Settlement→Réglé/Reçu). Le statut composite affiché est une projection (read-model) reconstruite des événements, jamais un champ traversant les contextes.

AD-22 Outbox transactionnel

Empêche
une notification perdue au recyclage Passenger ; un double-envoi ; une « transmission » silencieusement échouée.
Règle
Notification écrite dans un outbox, même transaction que le changement d'état. Relais après commit + clé d'idempotence. « Dépôt = transmission » satisfait par l'enregistrement durable. Les notifications persistées alimentent la cloche in-app (port de lecture). Matrice de routage FR30 portée ici.

Le cœur CRA

AD-6 Propriété & résolution du TJM

Empêche
deux sources de vérité du TJM ; des plages chevauchantes.
Règle
Projects est seul propriétaire des plages TJM (non chevauchantes). CRA résout via le port TjmResolver.resolveForDate(date), jamais en lecture directe.

AD-7 Machine à états possédée par l'agrégat

Empêche
un statut fixé hors du domaine ; une transition illégale ; deux CRA pour une même période.
Règle
Transitions = méthodes de l'agrégat (soumettre/valider/invalider) qui vérifient l'état de départ. Aucun use-case/repo ne fixe le statut. Les éditions de contenu en Brouillon/Invalidé sont auto-sauvées sans transition. Unicité : un CRA par (projet, période).

AD-8 Immuabilité + snapshot figé

Empêche
la modification d'un engagement signé ; trois montants divergents (CRA/PDF/facture).
Règle
À la validation, snapshot de forme fixe {montant_total_centimes, devise, tjm_appliqués[], jours[]} + horodatage + valideur. PDF et facture dérivent du snapshot, jamais recalculés. Aucune méthode de déverrouillage n'existe — l'absence est la garantie.

AD-9 Légalité (domaine) vs autorisation (application)

Règle
Le domaine garantit la légalité de la transition (état de départ). L'application garantit l'autorisation (rôle + organisation) via un port de politique.

Sécurité & cloisonnement

AD-10 Cloisonnement par appartenance au projet

Empêche
une fuite inter-orga ; qu'un CRA co-détenu fuite ou renvoie 404 au valideur.
Règle
Contexte d'acteur {userId, rôles-par-projet} = paramètre obligatoire de chaque use-case. L'accès dérive de l'appartenance au projet + rôle, jamais d'une égalité org_id = acteur.orgId (un CRA est co-détenu par les deux organisations).

AD-11 Désactivation, jamais suppression

Règle
Un membre ayant agi est désactivé, jamais supprimé : l'identité du signataire survit. Un membre retiré perd immédiatement l'accès (sessions invalidées).

AD-14 Auth ; identité-seule au BFF, rôles côté IAM

Empêche
un token exposé ; un NestJS non gardé ; une escalade via des rôles forgés/périmés.
Règle
argon2id + rate-limit login. Session cookie httpOnly persistée en MariaDB. Le BFF forge un JWT court portant uniquement userId — il n'autorise rien. IAM est la source de vérité des rôles, résolus à chaque requête. NestJS rejette toute requête sans le credential signé : la frontière de confiance est le secret, pas le réseau.

AD-23 Cycle de vie IAM

Règle
Toute organisation a ≥ 1 propriétaire actif (retrait du dernier interdit). Projects possède l'Invitation ; à l'acceptation, un use-case orchestre création de compte IAM + rattachement org/rôle.

Contrat d'API & données

AD-12 Contrat unique, client généré, anti-drift

Règle
DTO NestJS décorés → OpenAPI → client TS généré pour Nuxt. La génération est une tâche du pipeline Turborepo. Le front appelle l'API uniquement via le client généré.

AD-13 Validation à deux niveaux

Règle
Structurelle au bord (class-validator NestJS, jamais confiance au relais BFF) + métier dans les agrégats. La validation runtime fait foi côté NestJS.

AD-17 Représentation monétaire unique

Empêche
des erreurs d'arrondi ; deux contextes échangeant des montants sous des formes différentes.
Règle
VO Money = {centimes, devise:"EUR"}, jamais de flottant. Arrondi half-up à 2 décimales au niveau ligne, somme exacte. Ordre de calcul fixé.

AD-20 Statut calculé back, reflété front

Règle
Le statut (machine à états du domaine) est la source de vérité ; le front le reflète (pastille + libellé + icône, jamais la couleur seule) sans recalculer de transition. Jours fériés calculés côté serveur.

Facture, documents, front, ops

AD-15 Facture : dépôt = transmission, jamais de génération

Règle
Dépôt impossible tant que le CRA n'est pas validé. Dépôt = transmission (via outbox durable). PDF déposé validé au bord (MIME + taille). Remplacement possible tant que le règlement n'est pas confirmé. L'app ne génère jamais de facture (certification FR).

AD-16 Documents privés, PDF figé

Règle
Fichiers sur disque local hors webroot, jamais d'URL publique ; accès uniquement par endpoint authentifié + autorisé. PDF du CRA généré à la validation, immuable.

AD-18 Migrations au déploiement

Règle
Migrations MikroORM jouées explicitement à l'étape de déploiement, jamais au boot (process Passenger multiples).

AD-19 Accès données front via le BFF

Règle
Tout accès données passe par les server routes Nuxt (useFetch/$fetch). Aucun composant n'appelle NestJS ni ne manipule de token. État client = Pinia.

AD-24 Sauvegarde & reprise (données probantes)

Empêche
une perte irréversible de données signées sur un point de défaillance unique o2switch.
Règle
JetBackup o2switch + dump MariaDB périodique automatisé (incluant les fichiers), conservés hors o2switch. Restauration testée avant prod.

Différé

  • Isolation réseau réelle de NestJS — durcissement via VPS, plus tard.
  • Staging — local + prod seuls en v1.
  • Rôle intermédiaire + marge + confidentialité des prix (v2).
  • Module contrat collaboratif (v2 : sections, export Word/PDF, signature du contrat).
  • Synthèses / dashboards pluriannuels (v2).
  • Signature eIDAS avancée — Option A suffit en v1.
  • Conservation & RGPD — durée de rétention ; anonymisation vs AD-11 face au droit à l'effacement.
  • Observabilité — logs/métriques/alerting non outillés en v1.
  • MikroORM v7 — migration différée après stabilisation v1.

À réconcilier en amont

Deux divergences source connues, issues de décisions prises en coaching :
  1. Addendum PRD dit PostgreSQL → l'archi a basculé sur MariaDB (PostgreSQL o2switch = 9.6 EOL en voie de retrait). À mettre à jour dans l'addendum.
  2. Doc UX (draft) évoque un déverrouillage admin d'un CRA validé → contredit FR19 (immuable, jamais déverrouillable). À retirer du doc UX. La référence design y est aussi à corriger vers prototype-uiux-v2.
