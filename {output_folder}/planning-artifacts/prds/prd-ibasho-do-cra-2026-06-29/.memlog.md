- (decision) Voie coaching choisie.
- (decision) Admin = role VRAIMENT separe du prestataire (pas de fusion).
- (decision) Deux cotes = deux ORGANISATIONS (orga prestataire + orga cliente). Le CREATEUR du projet invite les membres des deux orgas via une interface d'invitation: il qualifie l'invite (cote prestataire ou cote client) + saisit l'email => email d'invitation envoye. Un meme utilisateur peut CUMULER valideur + payeur.
- (decision) Confirmations de reglement OPTIONNELLES et PARAMETRABLES: (a) payeur clique 'Regle' (date), (b) prestataire clique 'Reception confirmee'. Ni l'une ni l'autre obligatoire; depend du parametrage voulu par prestataire et client. Si active, les deux cotes voient l'avancement.

Permissions proprietaire: creer/configurer le projet, inviter+qualifier les membres des 2 orgas, gerer le parametrage, deverrouiller un CRA valide.
- (decision) Parametrage (dont confirmations de reglement) defini a la creation du projet MAIS modifiable a tout moment, gere par le createur/proprietaire du projet. v1: createur = prestataire (Arnaud).
- (decision) TJM borne dans le temps: reference = le CONTRAT. On definit un TJM de telle date a telle date (plusieurs periodes de TJM possibles par projet). Le montant d'un CRA = somme par jour de (fraction de jour x TJM applicable a cette date).
- (decision) Validation: le valideur INVALIDE le CRA EN ENTIER avec un commentaire OBLIGATOIRE expliquant pourquoi (pas de contestation jour par jour). Validation = globale.
- (decision) Periode du CRA PARAMETRABLE par projet, par defaut MENSUELLE.
- (decision) Notifications (corrige): CRA valide => notifie PRESTATAIRE seulement (le payeur voit l'info sur son interface mais N'EST PAS notifie). Facture deposee => notifie PAYEUR seulement. Regle invariant: la facture ne peut etre deposee QUE si le CRA est valide. Autres notifs confirmees (soumis=>valideur, refuse=>prestataire, regle=>prestataire, reception confirmee=>payeur+valideur).
